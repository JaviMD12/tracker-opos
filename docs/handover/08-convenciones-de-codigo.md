# 8. Convenciones de código a seguir

- **`app/models/` es un paquete**, no un `models.py` plano — un archivo por modelo (`marca.py`, `sesion_estudio.py`, `resultado_simulacro.py`...). Esto se ha confundido repetidamente entre sesiones (se ha pedido "actualizar `models.py`" varias veces cuando ese archivo no existe) — si ves esa instrucción, es casi seguro que se refiere a `app/models/`.
- **Nombres en español, sin acentos, en identificadores/tablas/variables** (`usuario_id`, `fecha`, `duracion_minutos`, `titulo_plaza`...). El contenido de texto visible al usuario (strings, prompts) sí lleva tildes en UTF-8 correctamente — si se ven rotas en una consola de Windows, es el codepage de la terminal, no un bug real.
- **Reusar el campo/patrón que ya existe antes de crear uno nuevo con otro nombre.** Ejemplos reales de esta ronda: se pidió crear `is_premium` y se reusó `Usuario.is_pro` (ya existía con el mismo propósito); se pidió `StudyLog` con `user_id` y se creó `SesionEstudio` con `usuario_id` para mantener consistencia.
- **Todo router de datos nuevo**: `Depends(get_current_user)` + `.filter(Modelo.usuario_id == current_user.id)`. Si además es una funcionalidad de pago, añadir `if not current_user.is_pro: raise HTTPException(403, ...)` — ver el patrón ya replicado en `convocatorias.py`, `tutor.py`, `simulacros.py`.
- **Errores de SDKs externos** (Stripe, OpenAI, Authlib, httpx): capturar la excepción base del SDK, envolver en `HTTPException(502, ...)`. Para las excepciones de `stripe`, preferir el import directo (`from stripe import SignatureVerificationError` / `stripe.SignatureVerificationError`) en vez de `stripe.error.X` — esa segunda forma dio un error real y confuso con la versión actual del SDK (ver [02-autenticacion-y-pagos.md](02-autenticacion-y-pagos.md)).
- **Inicialización perezosa** para todo lo que dependa de una API key externa o de un recurso costoso (Stripe, OpenAI, el vectorstore de Chroma) — nunca llamadas de red/disco al importar un módulo.
- **Nunca `innerHTML` con contenido generado por el LLM sin `DOMPurify.sanitize(marked.parse(...))` primero** — patrón usado en el chat del Tutor IA, el plan de estudio del Tablón y en cualquier render de Markdown nuevo.
- **Frontend es Vanilla JS puro** (`frontend/js/main.js`, un único archivo, sin build step, sin módulos ES, sin framework) — mantener ese estilo al añadir features nuevas.
- **Antes de dar por bloqueada/protegida una funcionalidad de pago, verificar el 403 real del backend**, no solo el estado visual del frontend (`localStorage`). El botón "Simular pago" (dev) nunca activa `is_pro` de verdad — ver [02-autenticacion-y-pagos.md](02-autenticacion-y-pagos.md).
- **Antes de reconstruir una UI ("vamos a añadir un mapa de calor/heatmap...")**, comprobar si ya existe algo parecido de una sesión anterior con datos mock — ha pasado más de una vez en esta ronda (el heatmap de actividad ya existía visualmente antes de conectarlo a datos reales). Reusar y conectar, no duplicar el contenedor.
